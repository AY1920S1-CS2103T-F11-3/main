Analyse Password

The following class diagram is the current structure of the Analyser component, using DictionaryAnalyser as an example:

API : TODO

The Analyser component,

  • can analyse the current list of Password object.

  • can output a summary security report for the list of Password objects.

  • can output a detailed security report for a specified Password object.

Proposed Implementation

The analyse password functionality is facilitated by the AnalysePasswordCommand, which makes use of various analysers. Each type of analyser implements Analyser, and implements the following operations:

  • Analyser#analyse(List<Password> passwords) — Analyses a certain aspect for every password in the list.

  • Analyser#outputSummaryReport() — Outputs an overview summary report for the list of Results created from analysing each password.

  • Analyser#ouputDetailedReport(Index index) — Outputs a detailed report from the result of a specified Password.

Given below is the example usage of how the the analyse functionality, using the DictionaryAnalyser, behaves at each step.

Step 1. The user inputs analyse into the command line. The Logic uses the PasswordBookParser to parse the user command, which results in a AnalysePasswordCommand which is executed by the LogicManager.

Step 2. The AnalysePasswordCommand calls the Model#getFilteredPasswordList() to obtain the current list of passwords, as well as #getRequiredAnalysers() to get the default list of analysers.

Step 3. AnalysePasswordCommand will invoke Analyser#analyse() of each analyser on every password in the list to produce the respective result. The Result object contains information of the password after analysis, such as the description of result. For comparison-based analysers, Match objects are created to store information about matches found by the analyser. Each Match contains details such as the match token, as well as the start and end index of the token with respect to the password.

Step 4. A report is compiled by stringify-ing the list of Result objects from each analyser and shown to the user.

Step 5. Should the user wish to view a more detailed report for a specific password, user may input analyse strong/<index> to view a more detailed analysis of the specified password.

Design Considerations

Aspect: How analyse/analyse strong executes

  • Alternative 1 (current choice): Always analyse the entire list of Password objects for every "analyse" command, even if the list of Passwords was unchanged.

    • Pros: Easy to implement, not required to check state if the current list of Passwords has been modified.

    • Cons: May have performance issues in terms of speed of the programme.

  • Alternative 2: Save in memory the result produced by the Analyser objects, and update result upon modification of list of Passwords (e.g. new passwords added/ passwords deleted.)

    • Pros: Performance of programme will be a lot faster.

    • Cons: Hard to implement. Have to keep track of state of the list of Password objects and check if the list has been modified from the last time they were analysed.

Data Encryption

Initialization and Validation

All data files of SecureIT are encrypted with a master password set by the user. The initialization and validation of the master password is handled by TestStorage.

The following sequence diagram shows how the master password is initialized when the user uses the app for the first time, and validated for subsequent uses of the app.

Note that the main components of the app (Storage, Ui, Logic, Model) can only be initialised with a correct master password.

The app does not store the master password itself. Instead, during initialization, the app encrypts a magic word using the password and stores it in the storage. For validation, the app tries to decrypt the stored magic word using the password given and compares if the original word is obtained. If the password given is correct, the decrypted magic word should be the same as the original one.
